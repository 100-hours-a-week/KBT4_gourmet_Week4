# 회고 및 AI 사용 기록

## 회고

이번 과제에서는 지난주에 설계했던 REST API를 실제 Spring Boot 코드로 구현해봤다
처음에는 DB와 JPA를 연결해서 구현하는 방식으로 접근했지만, 과제 참고사항을 다시 확인해보니 이번 과제의 핵심은 DB 연동이 아니라 Request / Response 구조와 Postman을 통한 API 통신 확인이라는 것을 알게됐다.

그래서 DB를 사용하지 않고 Java 코드 내부의 `MemoryStore`를 임시 저장소로 사용하는 방식으로 변경했다.

이번 과제를 진행하면서 조금 헷갈리고 어려웠던 부분은 이렇게 고민 했던게 맞는건가 싶기도 하지만 Spring Security를 쓰지 말라 해서 API 응답 통신 관련해서 인증 되지도 않는데 회원가입이나 로그인 부분 관련해서 API 응답을 받아야 하는 생각을 했었다.
원래는 인증정보 바탕으로 사용자를 식별해야 하지만 Spring Security를 쓰지 말라해서 그냥 MemoryStore에 저장된 정보를 가지고 왔다.
나중에 Spring Security나 DB 등 확장 가능성을 위해서 기능별로 패키지 구분을 했고 아직 S.O.L.I.D 원칙에 대해서 다 배우진 않았지만 최대한 유지보수 하는 단계, 리펙토링 하는 단계에서 보다 쉽게 하려고 클래스 별 책임을 하나로 잡고 했다.
예제 코드 기반으로 구조를 이해하고 비슷한 패턴을 User/Post/Comment에 직접 적용했다.
GPT는 막힌 부분을 질문하는 용도로 사용했으며 답변을 그대로 쓰기보다 과제 조건에 맞게 코드를 이해하면서 도움을 받았다.

지난주 케빈이 피드백 해준 부분도 다시 공부하면서 고칠건 고쳐놨다.
401은 인증 실패, 403은 권한 없음, 404는 데이터 없음, 500은 서버 내부 오류라는 기준을 잡고, 상태 코드와 message가 서로 다른 의미를 갖지 않도록 잘 정리해야 한다고 깨달았다.

댓글 API 설계에서도 처음에는 댓글 생성은 게시글 하위 자원으로 두고, 수정/삭제는 commentId만 사용하면 된다고 생각 했는데
하지만 피드백을 보고 댓글을 게시글 하위 자원으로 표현하기로 했다면 조회/수정/삭제도 `/posts/{postId}/comments/{commentId}`로 맞추는 것이 더 일관적이라는 것을 이해했다.

이번 과제를 통해 단순히 API 목록을 작성하는 것에서 끝나는 것이 아니라, 실제로 요청을 보내고 응답을 확인하면서 Method, URL, Request Body, Response Body, Status Code가 서로 어떤 관계를 가지는지 더 구체적으로 이해할 수 있었다.

다음주 과제도 이번주 처럼 미리 시작하여 집중하면서 할 것이다.

## AI 사용 기록

이번 과제에서 AI를 사용한 부분은 다음과 같습니다.

1. CRUD의 의미와 HTTP Method별 역할을 질문했습니다.
2. DB 없이 Controller와 DTO만 사용해 JSON 요청/응답을 처리하는 구조를 질문했습니다.
3. 지난주 REST API 설계 피드백 중 상태 코드와 message 불일치, 댓글 URL 일관성, 좋아요 등록 상태 코드 기준에 대해 질문했습니다.

코드 작성 과정에서는 먼저 UserController를 과제 조건에 맞게 작성하는 예시를 요청했다. 이 예시를 바탕으로 Controller에서 RequestDto로 요청을 받고, MemoryStore에 데이터를 저장한 뒤, ResponseDto로 응답을 반환하는 흐름을 이해했다. 이후 PostController와 CommentController는 UserController에서 이해한 구조를 바탕으로 직접 비슷한 패턴을 적용하며 구현했다.

AI 답변을 그대로 제출하기보다는, 과제 조건인 “DB 없는 가정”, “Controller 중심 구현”, “DTO 활용”, “Postman 요청/응답 테스트”에 맞는지 확인하면서 코드와 제출 내용을 정리했습니다.
